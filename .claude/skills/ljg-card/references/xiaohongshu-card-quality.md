# 小红书多卡质量规范

用于 `-m` 多卡生成小红书图卡时的内容判断、配色选择和叙事结构。目标不是固定风格，而是让每组卡片都先匹配内容气质，再复用稳定的信息结构。

## 1. 先选主题，不要默认暗绿

生成前先读文章标题、导语、一级痛点和目标平台。根据内容气质选择一个主题；如果同时命中多个主题，优先选择更能服务读者预期的那个。

| 内容气质 | 触发信号 | 背景 | 强调色 | 视觉语言 |
|---|---|---|---|---|
| CLI / 工程工具 | CLI、命令行、开发者工具、浏览器自动化、开源库、终端 | `#0E1A15` / `#101F18` | `#63C987` | 暗色网格、终端窗口、代码块、命令流向 |
| AI 产品 / 效率工具 | App、工作流、个人效率、提示词、Agent 产品、SaaS | `#F7F3EA` / `#FFF9EF` | `#D97757` | 暖纸感、柔和面板、产品截图式卡片、轻量箭头 |
| 研究 / 模型 / 数据 | 论文、模型、benchmark、数据集、实验、评测 | `#F4F8F6` / `#EDF4F1` | `#2D6A4F` | 克制浅底、数据标签、坐标网格、注释框 |
| 商业 / 趋势 / 创业 | 市场、融资、商业模式、趋势、公司、行业变化 | `#F8F4EA` / `#FEFCF7` | `#9A6A2F` | 商业简报、分层信息、细分隔线、少量金色点缀 |
| 风险 / 安全 / 反思 | 风险、失败、隐私、权限、安全、警惕、别神化 | `#171717` / `#211B1B` | `#C75D54` | 高对比、警示条、风险清单、暗红点缀 |
| 创作 / 内容工具 | 写作、创作、媒体、图文、视频、表达、审美 | `#FBF6EA` / `#F5EBDD` | `#B77946` | 编辑部纸感、标题排印、便签、手稿线 |

**规则：**
- 暗绿终端只在 CLI / 工程工具主题中使用。
- 每组卡片最多 1 个强调色；不要用 AI 紫蓝霓虹。
- 技术内容不等于暗色；产品、趋势、研究类技术内容优先用浅色主题。

## 2. 卡组叙事结构

技术和工具类文章优先用 6 张结构：

1. **封面钩子**：一句话说清「为什么值得看」，最好含痛点或反常识。
2. **痛点共鸣**：列出现有做法为什么翻车，避免直接堆功能。
3. **机制解释**：用结构图讲「它换了什么打法」。
4. **核心能力**：3-4 个能力块，每块用人话解释，不只堆术语。
5. **快速上手 / 使用场景**：如果文章有命令或步骤，就做步骤卡；否则做场景卡。
6. **适合谁 + 边界**：说明谁该用、谁不用、哪些限制要想清楚。

非工具类内容可替换第 5 张：
- 研究类：实验结论 / 数据解读
- 商业类：影响对象 / 机会判断
- 创作类：可复制方法 / 模板
- 风险类：行动建议 / 避坑清单

## 3. 每张卡的排版要求

- 标题必须 1 秒内读懂，不超过 2 行为佳；需要断行时手动控制节奏。
- 正文优先短句，不把原文段落硬塞进卡片。
- 至少 2 张卡要有视觉结构：流程、对比、步骤、网格、清单之一。
- 命令、数据、产品名用等宽或标签处理，不能和正文混成一团。
- 手机缩略图下仍要看清主标题；正文不要小于 27px（1080 宽画布）。
- Footer 保持统一，但不要喧宾夺主。

## 4. 质量自检

截图前逐项检查：

- [ ] 主题是否由内容决定，而不是默认暗绿终端？
- [ ] 首图是否有明确钩子，而不只是复述标题？
- [ ] 第 2 张是否解释了读者为什么需要这个东西？
- [ ] 是否至少有一张机制图或对比图？
- [ ] 术语是否配了人话解释？
- [ ] 是否保留了限制和适用边界？
- [ ] 命令和长英文是否没有被尴尬断行？
- [ ] 源 HTML / prompt 是否留底，方便复现？
